## 第一章：拆解招标文件 —— 先搞清楚在跟谁竞争

### 1.1 拿到标书的第一反应

2026 年 3 月 14 日早上，你面前摆着一份 1865 行的招标文件。项目名称是"钱塘江流域杭州部分取用水监测计量能力提升重点项目"，预算 95 万元，截止日期 2026-03-27。

大多数人拿到招标文件的第一反应是：赶紧开始写。1865 行那么多内容，不抓紧时间怎么来得及？

但这恰恰是错误的第一步。

**正确的第一步不是动笔，而是拆评分标准。**

为什么？因为标书不是作文比赛——不是你写得好就能赢。标书是一场有评分规则的考试，你得先看清楚试卷结构，才知道哪道题值多少分。

### 1.2 评分结构拆解

花 20 分钟把评分标准拆出来，发现了这样的结构：

| 分数类型 | 分值 | 你能控制吗 |
|---------|------|-----------|
| 技术标 | 44 分 | 能，全靠写 |
| 管理标 | 8 分 | 能，全靠写 |
| 价格分 | 30 分 | 不能，取决于报价策略和竞争对手 |
| 资质分 | 18 分 | 不能，取决于公司资质和过往业绩 |
| **合计** | **100 分** | **你能控制 52 分** |

这意味着：你写标书能争取到的最大空间是 52 分（技术 44 + 管理 8）。剩下的 48 分由价格和资质决定，你写得再好也拿不到。

### 1.3 1865 行中只有两段重要

整份招标文件 1865 行，但对标书写作真正重要的只有两段：

| 段落 | 行数 | 作用 |
|------|------|------|
| 采购需求 | 约 70 行 | 告诉你"甲方要什么"——项目背景、技术指标、交付要求 |
| 评分标准 | 约 230 行 | 告诉你"怎么给分"——12 项评分对应 12 章技术标 |

其他 1500+ 行是什么？资格条件、商务条款、合同模板、投标须知——这些要看，但不影响你怎么写技术标。

### 1.4 12 章对应 12 项评分

评分标准有 12 项，你的技术标就应该有 12 章，**一一对应，不多不少**。

```
评分第 1 项  →  第 1 章（项目背景与理解）
评分第 2 项  →  第 2 章（现状分析）
评分第 3 项  →  第 3 章（总体方案设计）
...
评分第 12 项 →  第 12 章（售后服务方案）
```

> **关键原则**：不要自己发明章节结构。评分标准是你的章节大纲。评委拿着评分标准逐条打分——如果你的章节和评分标准对不上号，评委找不到对应内容，你就拿不到那一项的分。这是 D3（预判评审）在标书场景下最基本的要求。

### 1.5 教学总结

拿到任何标书、报告、方案任务，**第一步不是动笔，而是"拆评分标准"**。

具体动作：

1. 找到评分标准/审查要点/合同要求（通常在招标文件的后半部分）
2. 列出每一项评分的分值和评分要素
3. 确定哪些分是你能控制的、哪些不能
4. 用评分标准反推章节结构——评分几项，你就写几章

这个动作花 20-30 分钟，但能让后续所有写作都有靶心。

## 第二章：三级优先级原则 —— 脚本 > Skill > CC 直接

### 2.1 这个原则怎么来的

标书写作过程中，你反复面对一个选择：这件事是让 CC 直接做，还是写个脚本来做？

比如"把所有 bullet point 转成段落"这件事——你可以告诉 CC"帮我把这些 bullet 改成段落叙述"，CC 也能做。但问题是：

- 12 章技术标加起来几万字，CC 一次处理不完
- CC 处理完你怎么验证？用眼睛看？
- 下次写标书又要重新告诉 CC 一遍
- 更要命的是，CC 调用 API 处理文本时，返回的 `<thinking>` 标签可能污染你的文件（这真的发生了）

所以你做了一个决策：能用脚本的不用 CC。然后进一步推导出了三级优先级原则。

### 2.2 三级优先级

| 优先级 | 方式 | 适用场景 | 钱塘江标书中的例子 |
|--------|------|---------|------------------|
| **1（最高）** | Python 脚本 | 确定性操作、可重复、需审计 | `bullet_to_paragraph.py`、`bid_standardize.py`、`table_name_check.py` |
| **2** | Skill（CC + 约束） | 需要 AI 判断但有框架约束 | `report-writing` Skill 的 4 维度审查、subagent 写作模板 |
| **3（最低）** | CC 直接对话 | 探索性、一次性讨论 | 讨论招标文件拆解策略、讨论章节分配方案 |

### 2.3 为什么这样排

**脚本（优先级 1）**：

- 可重复：同一个脚本跑 10 次，结果一模一样
- 可审计：出了问题看脚本代码就知道哪一步错了
- 不消耗 token：跑脚本不花 API 费用
- 不依赖 context：脚本不会因为上下文被压缩而"忘记"规则
- 可独立验证：`bid_standardize.py --check` 只报告问题不修改，你可以先看报告再决定是否 `--fix`

**Skill（优先级 2）**：

- 有约束不跑偏：`report-writing` Skill 里写了 D1-D4 四个维度的检查清单，subagent 不会忘记检查
- 可复用：下次写标书还是同一个 Skill
- 但仍消耗 token：每个 subagent 调用都要花 API 费用
- 但仍依赖 context：如果 Skill 内容太长，可能在 compact 后丢失细节

**CC 直接（优先级 3）**：

- 最灵活：可以讨论任何问题
- 最贵：每一轮对话都消耗 token
- context 有限：长对话后前面说过的内容可能被压缩掉
- 不可重复：同样的问题换个会话可能得到不同答案

### 2.4 一个判断标准

> **如果一个操作你会做第二次，就写成脚本。如果需要 AI 判断但有固定流程，就写成 Skill。只有真正一次性的讨论才直接跟 CC 聊。**

钱塘江标书中的实际分配：

| 操作 | 做了几次 | 用了什么 | 如果重来 |
|------|---------|---------|---------|
| 消除 bullet point | 1 次（但未来每个标书都需要） | 脚本 | 脚本（正确选择） |
| 结构标准化 | 1 次（但未来每个标书都需要） | 脚本 | 脚本（正确选择） |
| 写各章节内容 | 12 章 × 可能重写 | Skill + subagent | Skill（正确选择） |
| 讨论章节分配 | 1 次 | CC 直接 | CC 直接（正确选择） |
| 检查敏感词 | 1 次（但本应每次自动检查） | 人工发现 | **脚本 + Hook**（事后才意识到） |

最后一行是本次最大的教训——第八章会详细讲。

## 第三章：并行写作架构 —— 主会话只调度，写作派 subagent

### 3.1 为什么主会话不直接写

最直觉的做法是：打开 CC，一章一章地跟它说"帮我写第 1 章"、"帮我写第 2 章"。

这个做法在短文档上没问题，但在 12 章技术标上会出大问题：

1. **Context 污染**：主会话写完前 4 章后，上下文已经被前面的文字内容占满。当你开始写第 5 章时，CC 对第 1 章的记忆已经模糊了。
2. **无法回溯**：如果你发现第 1 章需要修改，但主会话已经走到第 8 章，你很难让 CC 回忆起第 1 章的写作背景。
3. **串行太慢**：12 章一章一章写，每章 30 分钟就要 6 小时。并行 3-4 个 subagent 可以压缩到 2 小时。
4. **调度视角丢失**：主会话一旦陷入具体写作，就看不到全局了——不知道哪些章节需要交叉引用、哪些数据需要统一。

> **核心原则**：主会话只做规划和验收，保持 context 干净。所有文字写作都派 subagent。

### 3.2 第一轮：3 个 subagent 并行写初稿

```
主会话（调度中心）
├── subagent-A：写第 1-4 章（项目背景、现状分析、总体方案、技术方案细节）
├── subagent-B：写第 5-6 章（设备选型、数据平台方案）
└── subagent-C：写第 7-8 章（实施方案、组织架构）
```

### 3.3 第二轮：补写和重写

第一轮完成后验收，发现部分章节需要重写：

```
主会话（验收 + 再分配）
├── subagent-D：重写第 1 章（改为漏斗结构：宏观政策→省级要求→杭州需求→本项目）
├── subagent-E：重写第 9 章（重难点分析，需要更具体的技术难点和解决方案）
├── subagent-F：写第 10 章（进度计划 + 甘特图）
├── subagent-G：写第 11 章（质量保障方案）
└── subagent-H：写第 12 章（售后服务方案）
```

### 3.4 每个 subagent 的 prompt 结构

每个 subagent 收到的 prompt 包含四部分，来自 `report-writing` Skill 的 subagent 写作模板：

**第一部分：评分标准**

只给这个 subagent 负责章节的评分标准，不给其他章节的。比如写第 3 章的 subagent 只看到评分第 3 项的内容。

```
你是技术报告写作专家。写第 3 章：总体方案设计。

**评分标准**：
（此处粘贴评分标准第 3 项的具体内容）
```

**第二部分：4 维度检查清单**

```
**4 维度检查**：
- D1 来龙去脉：数据和事情的前因后果讲清楚，每个结论有推导过程，不跳步
- D2 乙方立场与措辞：站乙方立场，委婉表达，风险抛给甲方/政策
- D3 报告结构：先现状后方案，汇总一一对齐，结论对应回前文分析
- D4 预判评审：写完假装自己是审查专家，检查能不能问倒自己
```

**第三部分：格式规范**

```
**格式规范**：
- 禁用词：确保、我们、我司
- 禁用格式：bullet point（用表格或段落替代）
- 数据标注来源：[数据来源：XX] 格式
- 输出为 Markdown 格式
```

**第四部分：素材路径**

```
**素材文件**：
- 招标文件采购需求：~/Work/zdwp/bid/requirements.md
- 评分标准：~/Work/zdwp/bid/scoring.json
- 项目区域资料：~/Work/zdwp/bid/area_info.md
```

### 3.5 为什么按章节分而不是按维度分

和审阅时一样的理由：一个 agent 负责一个完整章节，效果最好。

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **按章节分（实际采用）** | subagent-A 写第 1-4 章全部内容 | 一个 agent 理解完整上下文、章节内部自洽 | 单 agent 任务较重 |
| 按维度分（未采用） | subagent-D1 只管所有章节的溯源链、subagent-D2 只管措辞... | 每个 agent 专注一件事 | 4 倍 token 消耗、D1 agent 和 D2 agent 建议冲突、合并困难 |

> **按章节分的核心理由**：同一章节内部需要上下文连贯。如果第 3 章的 3.1 节介绍了项目区域概况，3.2 节的方案设计需要引用 3.1 的结论。按维度分的话，管 D1 的 agent 在检查 3.2 的溯源链时，不知道管写作的 agent 在 3.1 里写了什么。

## 第四章：MD 工作流目录链 —— 流水线思维

### 4.1 为什么不是一步到位

标书写作不是"subagent 写完就交付"。subagent 产出的初稿有大量问题：

- 到处是 bullet point（CC 天生爱用 bullet）
- 格式不统一（有的章节标题是 `##`，有的是 `###`）
- 没有评分标准引用（评委需要看到你在回应哪一项评分）
- 有些数据没有标注来源
- 表格没有表名

如果把所有后处理都塞在一个步骤里，出了问题你不知道是哪一步坏的。所以需要流水线——每一步只做一件事，中间产物可以单独检查。

### 4.2 四级目录链

```
md/（初稿——subagent 直接产出）
  ↓ bullet_to_paragraph.py
md_clean/（清洗稿——消除 bullet point）
  ↓ report_quality_check.py --bid --fix
  ↓ bid_standardize.py --fix
md_final/（终稿——结构标准化完成）
  ↓ md_docx_template.py
merged.docx（最终交付物）
```

### 4.3 每一步做什么

#### md/ -> md_clean/：消除 bullet point

```bash
python3 ~/Dev/scripts/scripts/document/bullet_to_paragraph.py md/ --output-dir md_clean/
```

**做什么**：把所有 bullet point（`-`、`*`、`1.`、`2.`）转为段落叙述或表格。

**为什么需要**：公文写作严禁 bullet point。但 CC（以及所有 LLM）天生倾向于输出 bullet list——因为训练数据中大量使用这种格式。不靠后处理脚本，你要手动改几百个 bullet。

**踩过的坑**：第一版 `bullet_to_paragraph.py` 使用 Claude API 做转换。API 返回的内容中包含了 `<thinking>` 标签（这是 Claude 的推理过程标记），直接污染了 Markdown 文件。结果是 195 个 bullet 处理完反而变成了 329 个"问题"——因为 `<thinking>` 标签也被错误计入了。

重写后加了两个关键改进：
1. 过滤掉所有 `<thinking>...</thinking>` 标签
2. 逐块处理 + 失败重试，不把整个文件一次性丢给 API

#### md_clean/ -> md_final/：质量检查 + 结构标准化

```bash
# 两个脚本串联执行
python3 ~/Dev/scripts/scripts/document/report_quality_check.py md_clean/ --bid --fix --output-dir md_final/
python3 ~/Dev/scripts/scripts/document/bid_standardize.py md_final/ --scoring scoring.json --fix
```

**做什么**：
- `report_quality_check.py --bid --fix`：检查禁用词、重复行、数据来源标注是否齐全
- `bid_standardize.py --fix`：6 项结构标准化（下一章详细讲）

**为什么分两个脚本**：因为它们的关注点不同。质量检查关注内容（"有没有用错词"），结构标准化关注格式（"标题层级对不对"）。分开后可以独立迭代——你改了质量检查的规则，不会影响结构标准化的逻辑。

#### md_final/ -> merged.docx：合并 + 转 DOCX

```bash
python3 ~/Dev/scripts/scripts/document/md_docx_template.py merged.md -t template.docx -o merged.docx
```

**做什么**：把所有章节的 MD 文件合并为一个 MD，然后用 ZDWP 样式模板转换为 Word 文档。

### 4.4 教学总结

> **每一步都是独立脚本，可以单独跑、单独验证。出了问题知道是哪一步坏的。**

这就是第二章"三级优先级"里"脚本优先"的实际体现。如果把所有后处理都写成一个大脚本或者全交给 CC 处理，出了问题你只知道"最终结果不对"，但不知道是 bullet 转换的问题、结构标准化的问题、还是 DOCX 转换的问题。

**流水线的另一个好处是可以中断和恢复。** 假设 `bid_standardize.py` 出了 bug，你修好 bug 之后只需要从 `md_clean/` 重新跑，不需要从头让 subagent 重新写初稿。

## 第五章：bid_standardize.py —— 6 项结构标准化

### 5.1 为什么需要专门的标书标准化脚本

12 个 subagent 各自写各自的章节，风格不可能完全统一。有的 subagent 用 `##` 做章标题，有的用 `#`；有的给三级标题编了号，有的没编。

手动统一格式？12 章，每章几千字，你要花一整天。

所以写了一个脚本，一次性处理所有结构问题。这个脚本是整个标书工作流的核心——它让 12 个 subagent 的产出看起来像一个人写的。

### 5.2 6 项检查/修复功能

| # | 检查项 | 做什么 | 为什么需要 |
|---|--------|--------|-----------|
| 1 | 标题级别校正 | `## 第X章` -> `# 第X章`，子标题顺延降级 | 12 章合并后标题层级必须统一 |
| 2 | 评分引用注入 | 从 `scoring.json` 读取评分标准，以 blockquote 写入章首 | 评委需要看到你在回应哪一项评分 |
| 3 | 编号标签清理 | 去掉"创新点一：""难点二："等前缀 | subagent 喜欢加编号前缀，但公文不用这种格式 |
| 4 | 数据来源标注提取 | `[数据来源：XX]` -> blockquote 格式 | 统一数据来源的展示方式 |
| 5 | 加粗标题提升 | 独占一行的 `**XX**` -> 对应层级标题 | subagent 有时用加粗代替标题，需要规范化 |
| 6 | 三级标题自动编号 | 无编号的 `###` -> `X.Y.Z` 格式 | 确保所有三级标题有层次编号 |

### 5.3 两种模式

```bash
# check 模式：只报告问题，不修改文件
python3 ~/Dev/scripts/scripts/document/bid_standardize.py md_final/ --scoring scoring.json --check

# fix 模式：自动修复所有问题
python3 ~/Dev/scripts/scripts/document/bid_standardize.py md_final/ --scoring scoring.json --fix
```

> **总是先 check 再 fix。** 先看报告了解有多少问题、什么类型，确认脚本理解正确后再让它自动修复。这和 `docx_format_check.py` 的 snapshot -> compare 思路一脉相承——先看快照，再做手术。

### 5.4 一个具体例子：评分引用注入

假设 `scoring.json` 中第 3 项评分标准是：

```json
{
  "chapter": 3,
  "title": "总体方案设计",
  "score": 6,
  "criteria": "方案设计合理性、技术路线清晰度、创新性"
}
```

脚本会在第 3 章的章标题后自动注入：

```markdown
# 第3章 总体方案设计

> **评分标准（6分）**：方案设计合理性、技术路线清晰度、创新性
```

为什么要把评分标准写进来？两个原因：

1. **写的人自查**：subagent（或你自己）写完这一章后，对照 blockquote 看一眼——"合理性讲了吗？技术路线清晰吗？创新性体现了吗？"没覆盖到的赶紧补。这是 D3（预判评审）的具体操作。
2. **审的人对照**：后续人工审阅时，直接对着 blockquote 检查内容覆盖度，不用反复翻招标文件。

### 5.5 正则表达式设计

脚本中有几组关键正则，值得记住它们的设计思路：

```python
# 编号标签正则——去掉 subagent 生成的冗余前缀
NUMBERED_LABEL_RE = re.compile(
    r'(创新点|难点|亮点|特点|重点|要点|优势|挑战)'
    r'[一二三四五六七八九十\d]+'
    r'[：:]'
)

# 加粗标题正则——识别独占一行的加粗文本
BOLD_HEADING_RE = re.compile(r'^\*\*(.+)\*\*$')

# 三级标题编号正则——匹配已有编号和未编号的三级标题
H3_NO_NUMBER_RE = re.compile(r'^###\s+(?!\d+\.\d+\.\d+\s)(.+)$')
```

这些正则是从 12 章初稿中观察到的实际问题提炼出来的——不是凭空设计的。每一条正则背后都有一个 subagent 犯过的具体错误。

## 第六章：图表生成系统 —— ASCII art 到 PNG

### 6.1 问题发现

12 章初稿审阅时发现了一个预料之外的问题：subagent 在 7 处需要图表的地方，生成了 ASCII 文本图。

比如第 10 章的进度计划，subagent 写了这样的"甘特图"：

```
任务名称          3月    4月    5月    6月
前期准备          ████
设备采购          ██████████
安装调试                 ██████████
系统集成                        ████████
验收交付                              ████
```

这在 Markdown 里看着还行，但转成 Word 或打印出来完全不可用——字符间距不对，排版会乱。

### 6.2 图表工具链架构

为此建立了一套数据驱动的图表生成系统：

```
chart_common.py（公共模块：中文字体、ZDWP 配色、统一输出）
├── chart_gantt.py（JSON 配置 → 甘特图 PNG）
├── chart_bar.py（JSON 配置 → 条形图/柱状图 PNG）
└── chart_flow.py（JSON 配置 → 架构/流程图 PNG）

chart_insert.py（扫描 MD 中 ASCII art → 替换为 ![图名](path)）
```

**关键设计：数据驱动。** 不同项目只需要改 JSON 配置文件，脚本不用改。

### 6.3 具体例子：甘特图

JSON 配置（`charts/gantt_schedule.json`）：

```json
{
  "title": "项目实施进度计划",
  "tasks": [
    {"name": "前期准备", "start": "2026-03", "end": "2026-03"},
    {"name": "设备采购", "start": "2026-03", "end": "2026-04"},
    {"name": "安装调试", "start": "2026-04", "end": "2026-05"},
    {"name": "系统集成", "start": "2026-05", "end": "2026-06"},
    {"name": "验收交付", "start": "2026-06", "end": "2026-06"}
  ]
}
```

生成命令：

```bash
python3 ~/Dev/scripts/scripts/document/chart_gantt.py charts/gantt_schedule.json -o charts/gantt_schedule.png
```

输出一张标准的甘特图 PNG，使用中文字体和 ZDWP 统一配色方案。

### 6.4 chart_common.py 的价值

公共模块解决的是"所有图表看起来像一套"的问题：

- **中文字体**：统一使用系统中文字体，避免乱码
- **ZDWP 配色**：蓝色主色 + 灰色辅助色，和公司其他文档一致
- **统一输出**：300 DPI、白色背景、统一边距

如果没有这个公共模块，三个图表脚本各自处理字体和配色，结果可能是甘特图用蓝色、条形图用绿色、流程图用红色——放在同一份标书里就很不协调。

### 6.5 chart_insert.py：自动替换

```bash
python3 ~/Dev/scripts/scripts/document/chart_insert.py md_final/ --charts-dir charts/
```

这个脚本扫描 MD 文件中的 ASCII art 块，自动替换为 `![图名](charts/xxx.png)` 格式。

它不是靠 AI 理解"这是一个甘特图"来替换的——而是靠事先在 MD 中打标记（比如 `<!-- CHART: gantt_schedule -->`），然后脚本找到标记、查找对应的 PNG 文件、做替换。

> **这又回到三级优先级**：确定性操作（找标记、做替换）用脚本，不靠 AI 判断。

## 第七章：表格标准化 —— 89 张表的批量处理

### 7.1 问题发现

在 `md_final/` 完成结构标准化后，做最终检查时发现了一个大问题：

- 89 个表格全部没有表名
- 22 个表格没有引导段落（就是光秃秃地放了一个表格，上面什么都没有）

在公文和技术标书中，表格的正确格式是：

```
引导段落（解释为什么需要这个表格、怎么看这个表格）
表X-Y 表格名称
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| ... | ... | ... |
```

而 subagent 产出的是：

```
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| ... | ... | ... |
```

没有表名、没有引导——读者看到表格不知道它在说什么、为什么出现在这里。

### 7.2 处理流程

**第一步：标记所有需要处理的表格**

```bash
python3 ~/Dev/scripts/scripts/document/table_name_check.py md_final/ --fix
```

脚本做两件事：
1. 在没有表名的表格上方插入 `[待命名]` 占位符
2. 在没有引导段落的表格上方插入 `TABLE_NEEDS_INTRO` 标记

**第二步：5 个 subagent 按章节并行填写**

```
主会话（分配任务）
├── subagent-1：填写第 1-3 章的表名和引导段落
├── subagent-2：填写第 4-5 章
├── subagent-3：填写第 6-7 章
├── subagent-4：填写第 8-9 章
└── subagent-5：填写第 10-12 章
```

每个 subagent 的 prompt 明确要求：
- 表名格式：`表X-Y 表格名称`（X 是章号，Y 是序号）
- 引导段落至少 80 字（来自 D1 来龙去脉的要求）
- 引导段落必须包含：为什么需要这个表格、依据什么标准、不这样做会怎样

### 7.3 踩坑：检测逻辑 bug

初版 `table_name_check.py` 的检测逻辑有问题——它只检查表格上方第一行是不是表名。但实际情况是，subagent 写完引导段落后，表名和表格之间可能隔了空行：

```
引导段落 balabala...

表3-1 水文监测站点一览表

| 站点名称 | 类型 | 经度 | 纬度 |
|---------|------|------|------|
```

初版脚本看到表格上方第一行是空行，就判断"没有表名"，于是又插了一个 `[待命名]`。

修复方式：改为**向上搜索 10 行**，在 10 行范围内找到表名格式（`表X-Y`）就判定有表名。

### 7.4 踩坑：表名位置顺序

5 个 subagent 填写完毕后，发现有 79 个表格的"引导段落"和"表名"顺序搞反了：

```
❌ 错误顺序：
表3-1 水文监测站点一览表
为全面掌握项目区域内水文监测站点的分布...
| 站点名称 | ...

✅ 正确顺序：
为全面掌握项目区域内水文监测站点的分布...
表3-1 水文监测站点一览表
| 站点名称 | ...
```

正确的公文格式是：引导段落 -> 表名 -> 表格。不是表名 -> 引导 -> 表格。

79 个表格手动调？不可能。写了 `fix_table_order.py` 批量修复。

### 7.5 教学总结

这一章的教训是：**表格标准化应该提前做，不应该等 89 张表都写完了才发现没表名。**

如果重来一次，应该在 subagent 写作模板中就明确要求：

```
**表格规范**：
- 每个表格必须有表名，格式为"表X-Y 表格名称"
- 表名前必须有引导段落（≥80字）
- 顺序必须是：引导段落 → 表名 → 表格
```

这样 subagent 从一开始就按正确格式写，后处理的工作量会少很多。

## 第八章：安全事故 —— subagent 泄露投标单位名称

### 8.1 发生了什么

这是整个标书项目中最严重的事故。

第七章的 5 个 subagent 在填写表格引导段落时，有的 subagent 写入了"杭州市水文水资源监测中心"——这是投标单位的名称。

技术标评审是匿名评审。如果评委在技术标中看到投标单位的名称，这份标书**直接废标**。95 万的标书，几天的工作，归零。

### 8.2 泄露范围

紧急全文搜索后发现：

| 泄露内容 | 出现次数 | 处理方式 |
|---------|---------|---------|
| "杭州市水文水资源监测中心" | 多处 | 替换为"市级水文监测部门"或"项目业主" |
| "采购人" | 110 处 | 替换为"甲方" |
| "杭州市水土保持中心" | 少量 | 全部清除 |

"采购人"为什么也要替换？因为招标文件里"采购人"就是指具体单位，在标书中使用这个词，审查阅读时会自然联想到具体单位。替换为"甲方"更安全。

### 8.3 根因分析

这是最重要的部分。不是为了追责，而是为了搞清楚：**为什么 subagent 会写入投标单位名称？**

**原因 1：subagent 的权限和主线程一样宽。**

subagent 可以使用所有工具——包括 WebSearch。它可能从网上搜索到了"钱塘江流域杭州部分"相关信息，从中获取了单位名称。也可能从项目目录中的其他文件中读到了单位名称。

你没有限制 subagent 的工具权限，所以它能读取的信息范围远超你的预期。

**原因 2：subagent 的 prompt 里没有明确禁止。**

prompt 中写了"禁用词：确保、我们、我司"，但没有写"禁止出现任何单位名称"。对于 CC 来说，"禁止出现任何单位名称"和"禁用词：确保"是完全不同量级的约束。你以为 CC "应该知道"标书不能出现单位名称，但 CC 不知道——它没有这个行业常识，除非你明确告诉它。

**原因 3：没有自动检测机制。**

即使 subagent 写入了单位名称，如果有一个 Hook 在每次文件写入后自动检查敏感词，你会在第一时间发现问题。但你没有 Hook。所以问题一直潜伏到你人工审阅时才被发现。

### 8.4 如何防止（结合六层框架）

回到六层架构的思路（详见 `Layer4-Hooks.md`），分三层防护：

**Layer 4（Hooks）—— 强制检测**

PostToolUse Hook：每次 subagent 执行 Edit 或 Write 后，自动检查敏感词。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "python3 ~/Dev/scripts/scripts/document/check_sensitive_words.py $FILE_PATH --wordlist bid_sensitive.txt",
        "description": "编辑/写入文件后检查标书敏感词"
      }
    ]
  }
}
```

敏感词清单（`bid_sensitive.txt`）应该包含：
- 投标单位名称
- 投标单位地址
- 联系人姓名
- 联系电话
- 竞争对手名称
- "采购人"等间接暴露词

**Layer 5（Subagents）—— 权限限制 + 明确禁止**

限制 subagent 的工具权限：禁用 WebSearch，避免 subagent 从网上获取单位信息。

在 prompt 中明确写禁止条款：

```
**绝对禁止出现以下内容**：
- 任何具体的单位名称（投标方、采购方、竞争对手）
- 任何具体的人名
- 任何联系方式（电话、地址、邮箱）
- "采购人"一词（用"甲方"替代）
如果你不确定某个名称是否可以出现，用"相关单位""相关部门"代替。
```

**Layer 6（验证）—— 最终扫描**

合并为 `merged.docx` 之前，跑一遍完整的敏感词扫描：

```bash
python3 ~/Dev/scripts/scripts/document/check_sensitive_words.py md_final/ --wordlist bid_sensitive.txt --recursive
```

### 8.5 这件事的根本教训

> **"约束设计"比"模型能力"更重要。**

CC 再聪明，没有约束就会犯错。你不能指望 CC "应该知道"标书不能出现单位名称——就像你不能指望一个新来的实习生"应该知道"公司的保密规定。你必须明确告诉它，而且最好用技术手段强制执行（Hook），而不是只靠"在 prompt 里写了一句话"。

这是 Layer 1-3（建议性的）和 Layer 4-6（强制性的）分界线的活生生案例：

| 层级 | 做法 | 可靠性 |
|------|------|--------|
| Layer 3（Skill） | prompt 里写"禁止出现单位名称" | CC 可能忘记、可能判断错误 |
| Layer 4（Hook） | 每次写文件后自动 grep 敏感词 | **系统级强制，CC 无法跳过** |
| Layer 6（验证） | 合并前跑敏感词扫描 | **兜底检查，最后一道防线** |

**如果你只能从这篇文档记住一件事，记住这个**：任何"绝对不能犯"的错误，都必须用 Hook 或脚本来防止，不能只靠 prompt。

## 第十章：回顾 —— 如果重来一次会怎么做

### 10.1 四件事应该提前做

**第一件：一开始就建 Hook。**

敏感词检查不该等出事才做。如果从第一个 subagent 开始写作时就有 PostToolUse Hook，110 处"采购人"泄露在第一处就会被拦截。

**具体做法**：在开始写标书之前，先配置好 `~/.claude/settings.json` 中的 Hook，把敏感词清单准备好。这个准备工作花 10 分钟，能省几个小时的紧急修复。

**第二件：subagent prompt 更严格。**

禁止事项要写得具体，不能只写"注意保密"。应该明确列出"不能出现什么"。

```
# 初版 prompt（太宽泛）
注意保密要求，不要泄露敏感信息。

# 改进版 prompt（具体明确）
绝对禁止出现以下内容：
- 任何具体单位名称（投标方、采购方、第三方）
- 任何具体人名
- 任何联系方式
- "采购人"一词（用"甲方"替代）
违反此规则的内容将导致废标。
```

**第三件：表格标准化提前做。**

不应该等 89 张表都写完了才发现没表名。应该在 subagent 写作模板中就包含表格规范（表名格式、引导段落要求、正确顺序）。

**第四件：整条流水线一键化。**

整个 `md/ -> merged.docx` 应该是一条命令，不需要手动分 5 步执行。可以写一个 `bid_pipeline.sh`：

```bash
#!/bin/bash
# bid_pipeline.sh — 标书后处理一键流水线
set -e

PROJECT_DIR="${1:-.}"
SCORING="${PROJECT_DIR}/scoring.json"
TEMPLATE="${PROJECT_DIR}/template.docx"

echo "Step 1/5: 消除 bullet point..."
python3 ~/Dev/scripts/scripts/document/bullet_to_paragraph.py \
  "${PROJECT_DIR}/md/" --output-dir "${PROJECT_DIR}/md_clean/"

echo "Step 2/5: 质量检查 + 结构标准化..."
python3 ~/Dev/scripts/scripts/document/report_quality_check.py \
  "${PROJECT_DIR}/md_clean/" --bid --fix --output-dir "${PROJECT_DIR}/md_final/"
python3 ~/Dev/scripts/scripts/document/bid_standardize.py \
  "${PROJECT_DIR}/md_final/" --scoring "${SCORING}" --fix

echo "Step 3/5: 图表替换..."
python3 ~/Dev/scripts/scripts/document/chart_insert.py \
  "${PROJECT_DIR}/md_final/" --charts-dir "${PROJECT_DIR}/charts/"

echo "Step 4/5: 敏感词扫描..."
python3 ~/Dev/scripts/scripts/document/check_sensitive_words.py \
  "${PROJECT_DIR}/md_final/" --wordlist "${PROJECT_DIR}/bid_sensitive.txt" --recursive

echo "Step 5/5: 合并 + 转 DOCX..."
python3 ~/Dev/scripts/scripts/document/md_docx_template.py \
  "${PROJECT_DIR}/merged.md" -t "${TEMPLATE}" -o "${PROJECT_DIR}/merged.docx"

echo "完成！输出：${PROJECT_DIR}/merged.docx"
```

### 10.2 做对了什么

也要承认这次做对的事情，下次继续保持：

**评分标准拆解先行。** 没有直接开写，而是花 20 分钟把评分结构搞清楚。这确保了 12 章的内容都指向了具体的评分项，不做无用功。

**主会话只调度，不写文字。** 主会话的 context 始终干净，可以做全局判断——比如发现第 1 章需要改为漏斗结构、发现第 9 章的重难点分析不够具体。如果主会话自己写了几章，context 就被文字内容占满了，做不了这种全局判断。

**三级优先级贯彻始终。** 确定性操作全部用脚本（bullet 转换、结构标准化、表名检查），需要 AI 判断的用 Skill 约束（subagent 写作模板），一次性讨论才直接聊。这让整个流程可重复、可审计。

**问题发现后立刻建工具。** 发现 89 张表没表名，不是手动一张张加，而是先写 `table_name_check.py`，再让 subagent 批量填写。发现 ASCII art 不能用，不是手动一张张替换，而是建了图表生成系统。**每次遇到重复性问题，都把解决方案固化为脚本**——这是三级优先级原则的自然延伸。

### 10.3 关键数字

| 指标 | 数值 |
|------|------|
| 总耗时 | 约 1 天 |

| 新建脚本 | 10 个 |
| 最终交付 | 12 章技术标 + 7 张图表 + 89 张命名表格 |

| 最大教训 | 约束设计 > 模型能力 |
