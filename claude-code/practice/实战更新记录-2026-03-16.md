## Ch03: 上下文工程 — Compact Instructions

### 理论回顾
Compact 会压缩对话历史以释放 context window，但压缩后容易丢失任务进度、决策上下文和文件结构认知。可以通过 Compact Instructions 指定压缩时必须保留的信息类型。

### 本次实操
在 `~/.claude/CLAUDE.md` 中新增了 `## Compact Instructions` 段落。痛点很明确：之前 compact 后经常丢任务进度（不知道做到哪了）、丢决策上下文（重新提议已经被否决的方案）、搞混文件结构（读过的文件忘了改过什么）。

**关键学习点**：Compact Instructions 保留的是**对话历史**，不是 CLAUDE.md 本身。CLAUDE.md 每轮对话都会重新加载，不需要在 Compact Instructions 里重复。所以这 6 条规则全部针对"对话过程中产生的临时信息"。

### 文件变更
- `~/.claude/CLAUDE.md` — 在"CC 使用规则"和"短规则"之间插入新段落

### 关键决策
- **位置选择**：插在"CC 使用规则"和"短规则"之间，因为 Compact Instructions 和会话管理逻辑紧密相关（上面就是"同一任务未完成时用 compact 继续"的规则）
- **6 条保留规则**：当前任务进度、关键决策（含被否决方案）、文件变更清单、错误与解决、方案引用路径、用户临时指令
- **token 开销**：约 200 tokens，在可接受范围内

### 踩坑与收获
这是 Ch11（CLAUDE.md 编写）"增量法"的实际应用——不是重写 CLAUDE.md，而是在合适的位置追加一个段落。实际写的时候才体会到"位置"的重要性：放错地方会让 CC 对规则的优先级判断出错。

```
## Compact Instructions

对话压缩时，必须保留以下信息：

1. **当前任务**：正在做什么、做到哪一步了、剩余步骤
2. **关键决策**：本次对话中做出的所有决策及理由（特别是被否决的方案，避免重新提议）
3. **文件变更**：读过/改过的文件路径列表，每个文件的变更要点
4. **错误与解决**：遇到的报错、踩坑及最终解决方式
5. **方案引用**：如果在执行某个方案文件，保留方案路径和当前执行位置
6. **用户临时指令**：本次对话中用户额外提出的要求或偏好
```

## Ch05: 工具设计 — AI 敏感词扫描脚本

### 理论回顾
工具设计的核心是让 CC 能调用外部能力（API、脚本），扩展它的行动边界。好的工具要有清晰的输入输出定义和错误处理。

### 本次实操
创建了一个 AI 敏感词扫描器，调用 Claude Haiku API 扫描标书 `.md` 文件，识别三类敏感词：组织名称（竞争对手）、不符合乙方立场的措辞（亟需/确保/我们）、项目名称交叉污染（A 项目文档出现 B 项目地名）。

### 文件变更
- `~/Dev/scripts/scripts/document/scan_sensitive_words.py` — AI 扫描主脚本（455 行），包含 API 调用、分块处理、幻觉过滤、交互式更新词表
- `~/Dev/scripts/raycast/commands/doc_scan_sensitive.sh` — Raycast 入口命令

### 关键决策
- **架构设计**：静态词表（Hook 用，快速确定性检查） + AI 扫描（维护词表，智能识别新敏感词）。两者互补：Hook 保证每次写入都检查已知词，AI 扫描负责发现未知词并补充到词表
- **模型选择**：用 Haiku（`claude-haiku-4-5-20251001`）而不是 Sonnet/Opus，因为敏感词识别是模式匹配任务，不需要深度推理，成本优化
- **幻觉过滤**：`verify_findings()` 函数验证 AI 返回的词是否真的出现在原文中，过滤 AI 编造的词条
- **分块策略**：8000 字符一块，在段落边界切割，避免截断句子导致误判

### 踩坑与收获
AI 工具的输出不能直接信任。加了幻觉过滤后，实际测试发现 AI 确实会"发明"一些原文不存在的敏感词。这验证了 Ch09（验证闭环）的核心理念：任何 AI 输出都需要验证步骤。

## Ch06: Hooks 机制 — PreToolUse 敏感词检查 Hook（核心实操）

### 理论回顾
Hooks 是在 CC 执行工具前/后自动触发的脚本，可以实现自动化的质量检查和安全防护。PreToolUse 在工具执行前触发，可以阻断不安全的操作。

### 本次实操
为 zdwp 项目创建了 PreToolUse Hook，在 CC 写入 `.md`/`.txt` 文件前自动检查内容是否包含敏感词。发现敏感词则阻断写入（exit 2），要求 CC 替换后重试。

### 文件变更
- `~/Work/zdwp/.claude/hooks/check_sensitive_words.sh` — Hook 脚本（107 行），从 stdin 读取 JSON、提取待写入内容、匹配敏感词表
- `~/Work/zdwp/.claude/settings.json` — 项目级 Hook 配置，matcher 为 `Edit|Write`
- `~/Work/zdwp/sensitive_words.json` — 敏感词配置文件，两类词：forbidden_words（6 个措辞类）+ project_isolation（2 个项目隔离类）

### 关键决策
- **PreToolUse 而非 PostToolUse**：这是本次最重要的设计决策。原笔记（Ch06）的设计是 PostToolUse，理由是"PreToolUse 时文件还没被修改，无法检查内容"。但这个理由**不准确**——PreToolUse 拿到的 `tool_input` 里就有即将写入的内容（Write 的 `content`、Edit 的 `new_string`），可以在写入前检查并阻断。PreToolUse 的优势：阻断后文件保持干净，不需要回滚
- **触发范围**：只在 `~/Work/zdwp/` 下生效（项目级 `settings.json`），不影响其他项目
- **文件过滤**：只检查 `.md`/`.txt`，跳过 `.py`/`.sh`/`.json` 等非标书文件
- **配置外置**：敏感词存在 `sensitive_words.json` 而非硬编码在脚本中，方便维护和被 AI 扫描器更新

### 踩坑与收获

**Windows 换行符问题**：脚本创建后执行报错，排查发现是 `\r\n` 换行符导致 bash 解析失败。用 `sed -i '' 's/\r$//' check_sensitive_words.sh` 修复。教训：CC 生成的 shell 脚本可能带 Windows 换行符，部署前要检查。

**对笔记的纠正**：在实操中发现原笔记的一个认知偏差——PreToolUse 并不是"拿不到内容"，而是拿到的是**即将写入的内容**。这正好说明实操比纯看笔记更有价值：只有动手做才能验证理论。

Hook 配置核心结构：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash /Users/tianli/Work/zdwp/.claude/hooks/check_sensitive_words.sh",
            "statusMessage": "正在检查敏感词..."
          }
        ]
      }
    ]
  }
}
```

## Ch07: Subagents 隔离 — Agent 模板更新

### 理论回顾
Subagent 通过 agent 模板文件定义角色和约束，实现任务隔离。好的模板要包含明确的输入/输出规范和质量检查清单。

### 本次实操
更新了 `bid-chapter-writer.md` 模板，新增三块规则，背景是实际业务中踩过的坑：89 张表无表名（评审扣分）、投标单位名称泄露事故（串标风险）。

### 文件变更
- `~/Work/zdwp/.claude/agents/bid-chapter-writer.md` — 新增表格规范、敏感词禁令、项目隔离三个 CRITICAL 段落

### 关键决策
- **表格规范**：每表必须有 `表X-Y 表名` 格式标题 + 引导段落 + 分析段落。这是从"89 张表无表名"的事故中总结的规则
- **敏感词禁令**：要求 agent 写作前加载 `sensitive_words.json`，与 Hook 形成双重保险——agent 自查 + Hook 兜底
- **项目隔离**：明确禁止 agent 使用记忆中其他项目的信息，只能用当前任务提供的素材路径
- **自查清单**：输出前必须逐条检查（禁用词、公司名称、其他项目地名、表格标题），体现 Ch09 验证闭环的思想

### 踩坑与收获
本次所有重活都派了 subagent 执行，主会话只做调度和审查。这正是 `feedback_use_subagents.md` 里记录的经验——主会话保持 context 干净，避免被具体实现细节填满导致 compact 后丢失主线。

## Ch09: 验证闭环 — Hook 测试验证

### 理论回顾
"定义清楚何为完成"是验证闭环的核心。任何自动化检查都需要测试用例覆盖正常和异常路径。

### 本次实操
手动构造 3 组 stdin JSON，模拟 PreToolUse 的输入格式，验证 Hook 脚本的行为。

### 测试用例

| 用例 | 输入 | 预期 | 实际 |
|------|------|------|------|
| 含敏感词的 .md 文件 | content 含 "亟需" | exit 2（阻断） | exit 2 |
| 干净内容的 .md 文件 | content 无敏感词 | exit 0（放行） | exit 0 |
| .py 文件（非目标类型） | file_path 为 .py | exit 0（跳过） | exit 0 |

### 踩坑与收获
测试让我确认了两件事：(1) Hook 的文件类型过滤逻辑正确，不会误伤代码文件；(2) 敏感词匹配是精确匹配（grep -F），不会因为正则特殊字符出问题。如果没有测试，这两个点只是"我以为"。
