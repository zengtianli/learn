# Layer 3：工作流层（Skills）

> **在六层架构中的位置**：第三层，告诉 Claude "怎么做"。把重复的多步骤工作流程固化成标准操作流程（SOP）。
>
> **前置阅读**：`Layer1-长期上下文.md`（Layer 1 定义"是什么"）、`Layer2-工具能力.md`（Layer 2 定义"能做什么"）
>
> **关联层**：Layer 3 使用 Layer 2 的工具来执行流程，受 Layer 1 的规则约束。Layer 4（Hooks）是 Layer 3 的"强制版"——Skill 是建议，Hook 是强制。

---

## 一、一句话理解

**Skill = CC 的"肌肉记忆"。** 如果 Layer 2 是"手脚"，Layer 3 就是"动作套路"。一个 Skill 定义了：遇到什么情况触发、按什么步骤执行、用哪些工具、输出什么格式。

你有了好工具（Layer 2），但如果每次都要从头告诉 CC"先提取文本、再分章节、再按 D1-D4 审阅、再写入修订标记、再验证格式"，那效率太低了。Skill 把这些步骤固化下来——你只需要说"帮我审阅这份报告"，CC 就自动走完整个流程。

## 二、三种 Skill 类型

### 2.1 检查清单型

**核心思路**："检查这些项"——CC 逐条过。

**特点**：
- 没有固定的执行顺序
- 每一项是独立的检查点
- 适合质量审查、规范校验

**我们的例子**：

| Skill | 内容 |
|-------|------|
| `report-writing` | 4 维度写作框架（D1 来龙去脉、D2 乙方立场与措辞、D3 报告结构、D4 预判评审） |
| `zdwp-structure` | PARA 目录结构规范 |
| `skill-standard` | Skill 编写的标准格式 |

**`report-writing` 的 4 维度清单**：

| 维度 | 名称 | 一句话 | 检查问句 |
|------|------|--------|----------|
| D1 | 来龙去脉 | 数据和事情的前因后果讲清楚了吗 | "审查者画个圈写'依据？'，我能回答吗？" |
| D2 | 乙方立场与措辞 | 责任给甲方/上级，我们不接 | "处长会皱眉吗？专家念出来会尴尬吗？" |
| D3 | 报告结构 | 先现状后方案，汇总一一对齐 | "每章结论单独排一列，能连成完整逻辑链吗？" |
| D4 | 预判评审 | 专家会问什么 | "如果我是审查专家，看到这页会提什么问题？" |

### 2.2 工作流型

**核心思路**："按这个顺序做"——CC 走流程。

**特点**：
- 有明确的执行步骤和顺序
- 步骤之间有依赖关系（Step 2 依赖 Step 1 的输出）
- 适合标准化的操作流程

**我们的例子**：

| Skill | 步骤数 | 内容 |
|-------|--------|------|
| `docx-review` | 5 步 | 格式快照 → 提取分章 → D1-D4 审阅 → 写入修订 → 格式验证 |
| `zdwp-risk-map` | 10 步 | QGIS 洪水风险图工作流 |
| `plan-first` | 3 步 | 先写方案再操作的规范 |

**`docx-review` 的 5 步 SOP**：

```
Step 0  格式快照（docx_format_check.py snapshot）
            ↓
Step 1  提取文本 → 分章节（docx_extract.py --split-chapters）
            ↓
Step 2  内容审阅（D1-D4 四维度）→ rules.json
            ↓
Step 3  写入修订标记（docx_track_changes.py review）
            ↓
Step 4  格式验证（docx_format_check.py compare）
```

这个 Skill 是你的系统中最完整的一个——触发条件明确、工具链完整、有单 Agent/多 Agent 两种模式、有 Agent Prompt 模板。

### 2.3 领域专家型

**核心思路**："你需要知道这些"——CC 作为领域顾问。

**特点**：
- 不定义执行步骤
- 提供领域知识、术语、业务规则
- CC 根据场景灵活运用这些知识

**我们的例子**：

| Skill | 内容 |
|-------|------|
| `zdwp-eco-flow` | 生态流量业务知识：Tennant 法、筛选链、报告章节结构、景宁经验 |
| `zdwp-water-rights` | 用水权知识 |
| `zdwp-context` | 水利公司项目上下文和常用命令 |

## 三、描述符优化原则

Skill 的描述符（SKILL.md 开头的描述）决定了 CC 什么时候触发它。

### 3.1 好描述 vs 坏描述

**坏描述**：
```
这个 Skill 用于处理文档相关的各种任务。
```

问题：太宽泛。CC 遇到任何文档任务都可能触发它，导致不该触发的时候也触发。

**好描述**：
```
当用户要求审阅 Word 文档时触发。按 5 步 SOP 执行：格式快照 → 提取分章 → D1-D4 审阅 → 写入修订 → 格式验证。
```

好在哪里：
- 触发条件具体："审阅 Word 文档"
- 概述了执行流程：5 步 SOP
- CC 能快速判断是否应该触发

### 3.2 描述符写作原则

| 原则 | 说明 |
|------|------|
| **短** | 控制在 2-3 行以内，CC 要快速扫描所有 Skill 描述来决定用哪个 |
| **精准** | 触发条件要具体，不要用"各种""相关"等模糊词 |
| **包含关键词** | CC 是通过关键词匹配来判断触发的。"审阅 Word"比"处理文档"更精准 |
| **概述流程** | 让 CC 在触发前就知道这个 Skill 会做什么 |

## 四、我们的核心 Skill 详解

### 4.1 report-writing（检查清单型）

**位置**：`~/.claude/skills/report-writing/`

**触发条件**：用户要求写报告或标书时。

**核心内容**：4 维度写作框架 + subagent 写作模板。

**4 维度框架**：

```
D1 来龙去脉（最重要）
├── 数据来龙去脉：来源、筛选链、趋势、分类、解读
└── 事情来龙去脉：背景、工作归口、承上启下

D2 乙方立场与措辞
├── 立场：不替甲方做决定、排除归因于政策
└── 措辞：软化绝对词、委婉化负面词

D3 报告结构
├── 信息顺序：先现状后方案、先全貌后细节
└── 汇总对齐：结论对应回前文分析

D4 预判评审
├── 假想追问：2-3 个专家可能的问题
└── 政策依据：排除结论引用政策、方法选择说明理由
```

**subagent 写作模板**：定义了 subagent prompt 的七要素（角色、任务、评分标准、4 维度清单、格式规范、素材路径、输出路径 + 禁止事项）。详见 `Layer5-Subagents.md` 第五章。

### 4.2 docx-review（工作流型）

**位置**：`~/.claude/skills/docx-review/`

**触发条件**：用户要求审阅 Word 文档时。

**核心内容**：5 步审阅 SOP + 多 Agent 模式。

**判断标准**：
- 文档超过 5 章或 50 页 → 用多 Agent 模式
- 文档较短 → 单 Agent 就够

**多 Agent 审阅分配**：

```
Lead Agent：
  1. 运行 Step 1 分章节
  2. 分配任务（按章节分，不按维度分）
  3. 每个 Agent 输出 rules_chN.json
  4. Lead 合并去重（review_summary.py --merge）
```

**为什么按章节分而不按维度分**：D1 来龙去脉要求 Agent 理解"上一节说了什么"才能判断承上启下是否到位。按维度分的话，Agent 缺乏完整上下文，判断会失准。

### 4.3 zdwp-eco-flow（领域专家型）

**位置**：`~/Work/zdwp/.claude/skills/zdwp-eco-flow/`

**触发条件**：涉及生态流量业务时。

**核心内容**：
- Tennant 法的计算方法和适用条件
- 生态流量核定的筛选链（总数 → 排除 → 最终核定对象）
- 报告章节结构
- 景宁项目的具体经验（戴欢 163 条修订的教训）

**设计亮点**：把业务知识（政策文件、计算方法、行业规范）和写作注意事项融合在一起。CC 在写生态流量报告时，既知道"怎么算"也知道"怎么写"。

## 五、全局 Skill vs 项目 Skill

| 位置 | 作用域 | 适用场景 | 我们的例子 |
|------|--------|---------|-----------|
| `~/.claude/skills/` | 所有项目 | 通用的方法论和流程 | report-writing、docx-review、plan-first、skill-standard |
| `项目/.claude/skills/` | 单个项目 | 项目特有的业务知识和流程 | zdwp-eco-flow、zdwp-risk-map、zdwp-context |

**分层原则**：
- 4 维度框架是通用的（报告和标书都用）→ 放全局
- Tennant 法是水利项目特有的 → 放项目级
- 审阅 SOP 是通用的 → 放全局
- 洪水风险图工作流是水利项目特有的 → 放项目级

## 六、反模式

### 6.1 Skill 职责重叠

**问题**：`report-writing` 里有 4 维度审阅框架，`docx-review` 里也引用了这个框架。如果更新了 4 维度定义，两边都要改，容易不一致。

**正确做法**：`docx-review` 应该显式声明"审阅维度来自 report-writing Skill"，而不是自己再写一遍。

### 6.2 触发条件太宽

**问题**：一个 Skill 的描述是"处理所有文档相关任务"——CC 遇到任何涉及文档的操作都可能触发它，即使只是简单的读取文件。

**正确做法**：触发条件要具体。"审阅 Word 文档"、"写技术报告"这样的描述，能让 CC 精准判断。

### 6.3 一次性流程写成 Skill

**问题**：把"迁移旧目录到新目录"写成 Skill。这种操作只执行一次，写成 Skill 浪费空间。

**正确做法**：只有重复 3 次以上的流程才值得写成 Skill。一次性操作直接在对话中说就行。

### 6.4 在 Skill 里写环境配置

**问题**：Skill 里写"先安装 python-docx"。

**正确做法**：环境配置不是工作流的一部分，应该在文档或 Hook 里处理。Skill 假设环境已经就绪。

### 6.5 操作步骤太细

**问题**：Skill 里写"打开终端 → cd 到目录 → 运行命令"。

**正确做法**：CC 知道怎么执行命令。Skill 应该描述"做什么"（运行格式检查），而不是"怎么按键"（打开终端、cd、输入命令）。

## 七、Skill 间的依赖关系

我们的 Skill 之间存在引用关系：

```
report-writing（4 维度框架）
    ↑ 被引用
docx-review（审阅 SOP 的 Step 2 使用 4 维度框架）
    ↑ 被引用
实战-写作质量体系.md（教学文件引用审阅 SOP）
实战-钱塘江标书全流程.md（标书项目复用 4 维度框架）
```

```
zdwp-eco-flow（生态流量知识）
    ↑ 被引用
zdwp-context（项目上下文引用生态流量 Skill）
```

**目前的问题**：这些依赖关系是隐式的——没有在 Skill 文件中显式声明。如果修改了 report-writing 的 4 维度定义，你不知道还有哪些 Skill 需要同步更新。

**改进方向**：在 Skill 的 SKILL.md 中加一段"依赖的其他 Skill"，显式声明引用关系。
