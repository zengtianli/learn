# Ch01: 底层运作原理

> Claude Code 不是编辑器插件，是一个在终端里跑的 Agent。理解它的核心循环，决定了你能不能把它用对地方。

---

## 理论框架

很多人第一次用 Claude Code 时，会把它当成"更聪明的自动补全"或者"终端里的 ChatGPT"。这两种理解都偏了。

CC 的本质是一个 **Agent**，它的工作方式是一个不断重复的三步循环：

```
收集上下文 → 采取行动 → 验证结果
     ↑                        │
     └────────────────────────┘
```

**收集上下文**：CC 读取 CLAUDE.md、Memory、Skills、MCP 工具定义，加上你给它的指令和项目文件，构建出"当前状况的理解"。这一步决定了 CC 知道什么、不知道什么。

**采取行动**：基于理解，CC 调用工具（Read/Edit/Bash/Grep 等）去做事——读文件、改代码、跑脚本、搜索内容。注意，CC 本身只能思考和写文本，所有与外部世界的交互都要通过工具完成。

**验证结果**：工具返回的结果进入 CC 的上下文，CC 判断"做对了吗？还需要继续吗？"。如果没做完或做错了，循环继续——再收集信息，再行动，再验证。

这个循环可能转很多圈。一个简单的任务可能只转一圈（读文件 → 改一行 → 确认），一个复杂任务可能转几十圈（搜索代码 → 理解结构 → 写方案 → 分步实现 → 逐个验证）。

**为什么理解这个循环很重要？** 因为它意味着：
1. **你给 CC 的上下文质量，直接决定行动质量。** 垃圾进垃圾出——CLAUDE.md 写得模糊、素材路径错误、需求描述不清，CC 的行动就会跑偏。
2. **CC 不是一次性输出结果，而是迭代逼近。** 它会犯错、会回头、会自我纠正。你要给它足够的空间去循环，而不是期望一步到位。
3. **每一步行动都消耗 context 窗口。** 工具调用的输入输出都进入上下文，窗口满了就要压缩（compact），压缩可能丢信息。所以让 CC 高效工作的核心是：减少无效循环，每一圈都有用。

## 实操案例

用三条主线工作流来看这个循环在实际工作中怎么转。

### 主线 A：写作质量体系——循环驱动的方法论涌现

这条主线的起点是一次报告对比（戴欢 vs 曾田力），终点是 4 维度写作框架 + 审阅 SOP + 工具链。

核心循环的体现：

```
收集上下文：CC 读取两个版本的报告，理解差异
     ↓
采取行动：拆解戴欢的 163 条修订，按类别分组
     ↓
验证结果：发现 7 成修订集中在"来龙去脉"（D1）
     ↓
继续循环：基于发现，提炼写作维度框架（最初 6 维度 → 最终 4 维度）
     ↓
再循环：用框架审阅报告，输出 19 条修订 + 19 条批注
     ↓
再循环：审阅时发现 CC 破坏了页眉页脚 → 催生格式保护工具
```

注意这里的循环不是"一次对话完成所有事"，而是**多次对话串联**。每次验证结果（发现新问题）触发下一轮行动（开发新工具）。这正是 Agent 循环的本质——不预设终点，根据实际情况持续迭代。

> 来源：`~/Learn/claude-code/00-总览与主线地图.md` 主线 A 第 2-7 节

### 主线 B：钱塘江标书——并行 Agent 的循环放大

一天内完成 12 章技术标初稿，核心策略是把循环"分发"给多个 subagent。

```
主会话的循环：
  收集上下文：拆解招标文件（100 分 = 52 分可控 = 12 章评分）
       ↓
  采取行动：3 个 subagent 并行，每个写 4 章
       ↓
  验证结果：验收发现第 1 章结构不好、89 张表格没表名、110 处单位名称泄露
       ↓
  继续循环：重写薄弱章节 + 5 个 subagent 填表名 + 紧急清理敏感词
```

这个案例展示了循环的一个关键特点：**验证环节暴露的问题，定义了下一轮行动的方向。** 如果没有验证，110 处单位名称泄露会直接导致废标。验证不是可选的收尾工作，是循环不可或缺的一环。

同时这也展示了"循环套循环"的结构——主会话有自己的大循环（调度→验收→修复），每个 subagent 内部也有自己的小循环（读素材→写初稿→自检→修改）。

> 来源：`~/Learn/claude-code/00-总览与主线地图.md` 主线 B

### 主线 C：VPS 基础设施——循环的基础设施层

这条主线看起来跟前两条无关，但它实际上是在构建"收集上下文"这一步的物理基础。

Syncthing 同步了 4 个目录（Work/zdwp、Personal/essays、Learn、Work/reports）到 VPS，中转站 API 让远程的 Edict 系统可以调用 Claude。这意味着：

- 远程 Agent 也能进入"收集上下文"的循环，因为它能访问到本地的工作文件
- 代理服务（Xray Reality 占 443 端口）确保了网络链路可用——没有网络，Agent 循环根本转不起来

**关键决策"443 给 Xray，Web 走 8443"**，本质上是一个"收集上下文"层面的优先级判断：代理流量的稳定性（不被检测）比 Web 服务的便利性更重要，因为代理一旦断了，整个 Agent 循环就断了。

> 来源：`~/Learn/claude-code/00-总览与主线地图.md` 主线 C

### 三条主线的关联：循环之间的依赖

三条主线不是孤立的。从循环的角度看它们的依赖关系：

| 依赖 | 实质 |
|------|------|
| 主线 A 的 4 维度框架 → 主线 B 的标书写作 | A 循环的验证产出，成为 B 循环的上下文输入 |
| 主线 A 的工具链 → 主线 B 的 DOCX 转换 | A 循环中开发的行动工具，被 B 循环直接复用 |
| 主线 C 的 Syncthing → 主线 A/B 的文件同步 | C 构建了"收集上下文"的物理通道 |

这说明 Agent 循环不仅在单个任务内运转，还在任务之间形成**循环的复用链**——一个循环的产出（框架、工具、基础设施），成为下一个循环的输入。

---

## 关键收获

- **CC 的核心循环是"收集上下文→采取行动→验证结果"，不是一次性输入输出。** 理解这一点，你才会重视上下文质量（CLAUDE.md 写清楚）、给 CC 迭代空间（不期望一步到位）、重视验证环节（验证驱动下一轮行动）。

- **上下文质量 > 模型智力。** 三条主线中最关键的不是 CC 有多聪明，而是你给它的素材有多好——戴欢的 163 条修订（高质量上下文）直接催生了 4 维度框架。招标文件的评分标准（精确上下文）让 subagent 知道每章该写什么。

- **验证不是可选项，是循环的组成部分。** 110 处单位名称泄露、89 张表格没表名、格式被破坏——所有重大问题都是在验证环节发现的。跳过验证等于让循环断裂。

- **循环可以嵌套和并行。** 主会话的大循环套 subagent 的小循环，3 个 subagent 并行各自转各自的循环。这个结构让 2 天内完成 14 个脚本 + 12 章标书 + 4 项 VPS 服务成为可能。

- **基础设施是循环的前提。** 网络、同步、API 中转这些"不起眼"的工作，实际上是在保障 Agent 循环能顺利运转。循环断在任何一个环节（网络断了、文件不同步、API 不可用），整个工作流都会停滞。
