# Ch02: 概念边界

> 8 个核心概念各管各的事，搞混了就会把力气用错地方。这章画清楚边界，知道什么该放在哪里。

---

## 理论框架

Claude Code 的能力体系由 8 个概念组成。它们经常被混用——比如把应该写成 Hook 的规则塞进 CLAUDE.md，或者把 Skill 能干的事交给 MCP。搞清楚每个概念的职责边界，是高效使用 CC 的前提。

### 8 个概念一句话定义

| 概念 | 一句话 | 加载方式 |
|------|--------|---------|
| **CLAUDE.md** | CC 的长期记忆和工作手册，每次会话自动加载 | 始终加载 |
| **Memory** | 跨会话持久化的结构化记忆，CC 可以自己写入 | 始终加载 |
| **MCP** | 外部工具协议，让 CC 连接第三方服务 | 连接时加载 |
| **Tools** | CC 内置的原子操作（Read/Edit/Bash/Grep 等） | 始终可用 |
| **Skills** | 按需加载的工作流模板，用 `/` 触发或 CC 自动匹配 | 按需加载 |
| **Hooks** | 确定性拦截层，在 CC 执行动作时自动运行你的脚本 | 始终生效 |
| **Subagents** | 隔离的子进程，拥有独立的 context 窗口 | 按需创建 |
| **Rules** | `.claude/rules/` 下的条件加载规则文件 | 始终加载 |

### 概念之间的关系

这 8 个概念不是平级的，它们构成了一个分层体系：

```
Layer 1（是什么）：CLAUDE.md + Memory + Rules
Layer 2（能做什么）：Tools + MCP
Layer 3（怎么做）：Skills
Layer 4（强制执行）：Hooks
Layer 5（并行隔离）：Subagents
```

越往上层，越偏"建议"；越往下层，越偏"强制"。CLAUDE.md 里写"禁止出现投标单位名称"，CC 可能忘记。Hook 里写同样的检查，CC 无法绕过。

### 典型误用场景

理解边界最好的方式是看"放错地方"会怎样：

| 误用 | 正确做法 | 为什么 |
|------|---------|--------|
| 把"标书禁止出现单位名称"写进 CLAUDE.md | 写成 Hook（PostToolUse 敏感词检查） | CLAUDE.md 是口头嘱咐，长对话中可能被压缩掉；Hook 是门禁，每次写文件都会检查 |
| 把审阅 SOP 的 5 个步骤写进 CLAUDE.md | 写成 Skill（docx-review） | CLAUDE.md 每次都加载，浪费 token；Skill 按需加载，只在审阅时触发 |
| 给每个脚本都注册 MCP Server | 通过 Skill 告诉 CC 脚本路径和参数 | 每个 MCP Server 约 4-6K token 固定开销；Skill 只在触发时加载 |
| 用 CLAUDE.md 写"遇到 Python 文件先 lint" | 写成 Hook（PostToolUse Python 语法检查） | CLAUDE.md 依赖 CC "记住"并主动执行；Hook 自动触发，不依赖 CC 的判断力 |
| 把项目特有规则放全局 CLAUDE.md | 放项目级 CLAUDE.md | 全局规则对所有项目生效，不该让水利项目的规则影响其他项目 |

## 实操案例

### CLAUDE.md：短、硬、可执行

全局 CLAUDE.md（`~/.claude/CLAUDE.md`）大约 2K token，包含协作模式、行为规范、技术栈规则、路径规范。

关键经验是：**CLAUDE.md 的每一行都在消耗 context 窗口。** 写进去的每条规则，都应该值得在每个会话中被 CC 读到。

好规则 vs 坏规则的对比（来源：`~/Learn/claude-code/Layer1-长期上下文.md` 第二章）：

```
好："技术栈用 Streamlit，不用 React"     → 没有歧义
坏："我们的技术栈比较灵活"               → CC 不知道该选哪个

好："Python 用 /Users/tianli/miniforge3/bin/python3" → 绝对路径
坏："用系统 Python"                      → 哪个系统 Python？

好："任务超 5 分钟必须拆分"               → 量化标准
坏："大任务建议拆分"                     → 多大算大？
```

经验法则：如果一条规则在 3 个以上的会话中都要用到，就写进 CLAUDE.md。只用一次的，直接在对话中说。

### Memory：CC 自己维护的记忆

Memory 文件（`~/.claude/projects/-Users-tianli/memory/MEMORY.md`）存的是跨会话的关键决策和偏好——工作追踪系统的设计方向、模型选择策略、OA 秘书系统的数据源等。

与 CLAUDE.md 的分界线：**CLAUDE.md 放规则（CC 必须遵守的指令），Memory 放记忆（CC 需要记住的决策和偏好）。** 规则是硬约束，记忆是参考信息。

### Tools：CC 的手脚

CC 内置的 Tools（Read/Edit/Write/Bash/Grep/Glob 等）是原子操作。用户的 30+ 个 Python 脚本通过 Bash 工具调用，本质上是扩展了 CC 的工具库。

脚本作为工具的设计原则（来源：`~/Learn/claude-code/Layer2-工具能力.md`）：

- **做一件事**：`docx_extract.py` 只管提取，`docx_format_check.py` 只管格式
- **名字自解释**：`{领域}_{动作}.py`，CC 通过名字就能猜出功能
- **双模式**：`--check`（只看）和 `--fix`（修复），先 check 确认再 fix 执行
- **输出面向 CC 优化**：错误信息告诉 CC "该怎么修复"，不只是"出了什么错"

三级优先级原则：

```
优先级 1：Python 脚本  — 确定性、可重复、不消耗 token
优先级 2：Skill（CC + 约束）— 需要 AI 判断但有框架约束
优先级 3：CC 直接对话  — 最灵活但最贵、最不可控
```

### Skills：CC 的肌肉记忆

Skills 分三种类型（来源：`~/Learn/claude-code/Layer3-工作流Skills.md`）：

| 类型 | 特点 | 实例 |
|------|------|------|
| 检查清单型 | 没有固定执行顺序，逐条过 | `report-writing`（4 维度框架） |
| 工作流型 | 有步骤和依赖关系 | `docx-review`（5 步审阅 SOP） |
| 领域专家型 | 提供领域知识，CC 灵活运用 | `zdwp-eco-flow`（生态流量业务知识） |

全局 Skill（`~/.claude/skills/`）放通用的方法论，项目级 Skill（`项目/.claude/skills/`）放项目特有的业务知识。分层原则：4 维度框架是通用的放全局，Tennant 法是水利特有的放项目级。

### Hooks：门禁而非口头嘱咐

Hooks 是六层架构中的 Layer 4，核心特征是**确定性**——不依赖 CC "记住"或"判断"，每次触发条件满足就执行。

最有教学意义的案例是标书敏感词检查（来源：`~/Learn/claude-code/Layer4-Hooks.md` 第四章）。钱塘江标书中 subagent 写入了 110 处投标单位名称，如果有 PostToolUse Hook，写入瞬间就会被发现。

三层防护的叠加关系：

```
CLAUDE.md："标书中禁止出现单位名称"   → 防护等级 ★★☆☆☆（可能被忘记）
Skill：审阅步骤中包含敏感词检查      → 防护等级 ★★★☆☆（只在审阅时检查）
Hook：每次 Edit/Write 后自动扫描     → 防护等级 ★★★★★（写入即检测）
```

判断标准：如果你能用 `grep` 或 `python -m py_compile` 检查出来的问题，适合用 Hook。需要"理解"才能判断的问题，不适合。

### Subagents：隔离 > 并行

Subagent 的核心价值不是快，是 context 隔离（来源：`~/Learn/claude-code/Layer5-Subagents.md`）。

主会话有 200K context。如果在主会话里连续写 12 章，写到第 7 章时前面的章节已经被压缩，写到第 12 章时第 1 章的背景可能已经丢了。用 subagent，每个 agent 有独立的 200K 窗口，主会话只存调度信息，始终保持干净。

权限控制是最大的改进点。subagent 默认权限和主会话一样宽——能读任何文件、能上网搜索、能执行命令。每次派 subagent 前要问：需要联网吗？需要写文件吗？需要执行命令吗？不需要的权限一律不给。

### Rules：CLAUDE.md 的拆分版

`~/.claude/rules/` 目录下的每个 `.md` 文件都会自动加载。当 CLAUDE.md 膨胀到超过 3K token，可以按主题拆分到 rules 目录。目前规则量不大，CLAUDE.md 已经够用。

## 关键收获

- **每个概念有明确的职责边界。** CLAUDE.md 管规则、Memory 管记忆、Tools 管操作、Skills 管流程、Hooks 管强制检查、Subagents 管隔离并行。把东西放对地方，比把东西写得漂亮重要得多。

- **"该用 Hook 还是 CLAUDE.md"的判断标准：确定性。** 能用模式匹配（grep）检查的 → Hook。需要语义理解的 → CLAUDE.md 或 Skill。越不可接受的错误（废标级），防护层次要越硬（Hook > Skill > CLAUDE.md）。

- **token 成本意识决定了"放哪里"。** CLAUDE.md 和 Memory 是固定开销（每次会话都付出），Skill 是按需加载，MCP 每个 Server 约 4-6K token。所以：常用规则放 CLAUDE.md（值得每次加载），重复流程放 Skill（触发时才加载），脚本通过 Bash 调用（零 token 开销）。

- **三级优先级是底层设计哲学。** 脚本 > Skill > CC 直接——能用脚本固化的不靠 Skill，能用 Skill 约束的不靠 CC 临场发挥。这个原则贯穿了所有 8 个概念的使用方式。

- **概念之间有依赖关系，不是各自独立的。** CLAUDE.md 的规则约束 Skill 的行为，Skill 引用 Tools 来执行流程，Hooks 对 Subagent 同样生效。理解这些依赖关系，才能设计出完整的工作体系。
