# Ch10: 高频命令与工程意义

> 了解这些命令的"为什么"比"怎么用"更重要。每个斜杠命令背后都是一个工程决策点。

---

## 理论框架

Claude Code 的交互式命令不只是快捷操作，它们直接映射到 LLM 工程的核心概念。

### 上下文管理三件套

这三个命令管的是同一件事：context window（约 200K tokens）的使用效率。

| 命令 | 做什么 | 工程意义 |
|------|--------|---------|
| `/context` | 查看当前上下文占用 | 可视化你的 token 预算消耗情况，判断何时该压缩或重启 |
| `/clear` | 清空对话，重置上下文 | 释放 context window，但不等于新会话——CLAUDE.md 和 Memory 仍然加载 |
| `/compact` | 压缩当前对话，保留关键信息继续 | 在不丢失任务上下文的前提下腾出空间，是长任务的生命线 |

关键区别：`/clear` 是"推倒重来"，`/compact` 是"轻装继续"。选择哪个取决于你是在切换任务还是继续当前任务。

### 资源与模型管理

| 命令 | 做什么 | 工程意义 |
|------|--------|---------|
| `/cost` | 查看当前会话的 token 消耗和成本 | 量化你的使用成本，发现哪些操作特别昂贵 |
| `/model` | 切换模型（Opus/Sonnet/Haiku） | 按任务复杂度选模型：简单任务用 Haiku 省成本，复杂任务用 Opus 保质量 |
| `/mcp` | 管理 MCP 服务器连接 | 扩展 CC 的工具能力，但每个 MCP Server 约消耗 4-6K tokens 的固定开销 |
| `/memory` | 管理持久化记忆 | 查看和编辑跨会话记忆，确保 CC 在新会话中记住关键决策 |

### 非交互模式（Headless）

Claude Code 不只是一个交互式终端。通过 `-p` 参数，它变成一个可以嵌入脚本和自动化流程的工具：

```bash
# 单次执行
claude -p "分析这个文件的性能瓶颈"

# 管道输入
cat error.log | claude -p "找出这些日志中的异常模式"

# 在 CI/CD 中使用
claude -p "检查这次 PR 的代码质量" < diff.txt
```

非交互模式的工程价值在于：**把 AI 能力从"人机对话"扩展到"系统间调用"**。定时任务、批量处理、Hook 触发——都可以通过 `-p` 模式实现。

## 实操案例

### 会话管理的实践规则

在日常使用中，我形成了明确的会话管理策略，记录在全局 CLAUDE.md 里：

```
# 来源：~/.claude/CLAUDE.md
- 会话管理：切换主题或任务完成时，重启新会话；同一任务未完成时用 compact 继续
- 记忆靠文件：关键决策写入 CLAUDE.md/MEMORY.md/方案 MD，不依赖会话 context
- 方案先行：大任务先写方案存档，再开新会话执行
```

这套规则的核心逻辑是：**context window 是宝贵资源，不要让无关内容占据它**。切换主题时用 `/clear` 或重启，因为旧主题的上下文会污染新任务的推理质量。同一任务用 `/compact` 继续，因为压缩后的摘要保留了关键决策链。

### 模型选择策略

```
# 来源：~/.claude/projects/-Users-tianli/memory/MEMORY.md
- 模型选择策略：简单任务用 Haiku（成本优化），复杂任务用 Sonnet/Opus
```

实践中的判断标准：如果任务主要是格式转换、信息提取——用 Haiku。如果任务涉及方案设计、多维度分析——用 Opus。`/model` 命令让你可以在同一个会话中按需切换，不必重启。

### 非交互模式：脚本集成

我的工作流中有一个典型的非交互使用场景——对话导出工具：

```
# 来源：~/.claude/projects/-Users-tianli/memory/MEMORY.md
- 脚本：~/Dev/scripts/scripts/secretary/export_cc_sessions.py
- Raycast：~/Dev/scripts/raycast/commands/sec_export_sessions.sh
```

CLAUDE.md 中的 Raycast 集成规则要求每个实用脚本都配对应的 Raycast Command。这意味着 CC 的能力可以通过 Raycast 的快捷键一键触发——底层走的就是非交互模式的思路：把 AI 操作封装成"按一下就执行"的原子动作。

### 环境配置的显式声明

```
# 来源：~/.claude/CLAUDE.md
Python: /Users/tianli/miniforge3/bin/python3
```

CLAUDE.md 中写了 Python 的绝对路径，而不是写"用系统 Python"。这条规则看起来简单，实际上解决了一个频繁出现的问题：macOS 上有多个 Python（系统自带的、Homebrew 装的、miniforge3 的），如果不指定绝对路径，CC 可能用错解释器，导致包找不到或版本不对。**环境配置的显式声明是减少调试时间的最有效手段之一。**

### Token 成本的实际感知

在 Layer1 笔记中做过一次固定开销的估算：

```
# 来源：~/Learn/claude-code/Layer1-长期上下文.md
固定开销（每次会话）：
├── 系统指令                    ~8K
├── Skill 描述（25+ 个）        ~5K
├── MCP 工具定义                ~2K
├── 全局 CLAUDE.md              ~2K
├── 项目 CLAUDE.md              ~1.5K
├── Memory 文件                 ~1.5K
└── 合计                       ~20K

实际工作可用空间：~180K
```

这组数据让 `/cost` 和 `/context` 不再是抽象概念。每个 MCP Server 占 4-6K tokens——这意味着加 5 个 MCP Server 就要多花 25K tokens 的固定开销，可用空间从 180K 缩到 155K。`/context` 让你看到这个代价，`/cost` 让你看到这个代价换算成多少钱。

## 关键收获

1. **`/compact` 是长任务的核心命令**。同一任务未完成时不要重启会话，用 `/compact` 压缩后继续。但要搭配 Compact Instructions 使用——告诉 CC 压缩时必须保留什么，否则关键规则可能被压掉。

2. **会话管理的本质是 context window 管理**。切换主题用新会话（避免上下文污染），继续任务用 `/compact`（保留决策链），关键信息靠文件持久化（不依赖会话记忆）。

3. **模型选择是成本-质量的权衡决策**。`/model` 命令让你在同一会话中按需切换——不是所有步骤都需要 Opus 级别的推理能力。

4. **非交互模式把 CC 从工具变成基础设施**。通过 `-p` 参数和管道输入，CC 可以嵌入脚本、Hook、定时任务——这是从"手动交互"到"系统自动化"的关键一步。

5. **环境配置必须显式声明**。绝对路径、具体版本、明确的技术栈选择——一条命令能查证的事情靠猜测下结论，是 CLAUDE.md 禁止行为清单里的一条，背后的道理是：CC 不会读心，你必须告诉它。
